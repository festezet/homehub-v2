🧠 Les 4 Disciplines du Prompting

Framework d'optimisation pour l'utilisation de Claude Code et des agents AI

📖 Qu'est-ce que les 4 Disciplines ?

Les 4 Disciplines du Prompting sont un framework structuré pour optimiser l'utilisation d'agents AI comme Claude Code. Elles permettent de passer d'une utilisation ponctuelle et improvisée à une collaboration systématique et efficace.

🎯 D1 - Prompt Craft (Individuel)

Art d'écrire un bon prompt ponctuel. C'est le niveau de base : savoir formuler une demande claire et précise pour obtenir le résultat souhaité. Pas de fichier ou système, juste la compétence de bien communiquer avec l'IA.

Ex : Savoir structurer une demande avec exemples et garde-fous - "table stakes"

📦 D2 - Context Engineering (Infrastructure)

Optimiser ce qui est chargé automatiquement. Les fichiers dans .claude/ sont chargés à chaque session. D2 consiste à auditer et optimiser ces fichiers pour garder seulement l'essentiel et réduire la "dette de contexte".

Ex : "99.98% du contexte n'est PAS ton prompt" - optimiser le reste

🧭 D3 - Intent Engineering (Stratégique)

Encoder les préférences et règles de décision. Au lieu de répéter les mêmes consignes à chaque session, on encode dans intent.md : trade-offs ("Fonctionnel > Parfait"), arbres de décision, triggers d'escalation.

Ex : Klarna 2.3M conversations - optimiser vitesse au lieu de satisfaction = echec

📋 D4 - Specification Engineering (Organisationnel)

Documenter pour exécution autonome. Pour les tâches complexes (>1h), on rédige une spec structurée que Claude ou un autre agent peut exécuter de A à Z sans avoir besoin de demander des clarifications.

Ex : Spec 11 min → agent autonome → 1 semaine en 1 matinee (ecart 10x)

💡 Pourquoi ce Framework ?

  • Réduction du bruit : Moins de tokens chargés = contexte plus dense et pertinent
  • Cohérence : Les règles sont encodées une fois, appliquées systématiquement
  • Autonomie : Claude peut prendre des décisions alignées sans demander confirmation à chaque fois
  • Scalabilité : Framework utilisable avec 5-10 sessions par jour sans dégradation
  • Capitalisation : Chaque session améliore le système pour les suivantes

💡 Exemples (issus de la video)

"99.98% du contexte n'est PAS ton prompt"

La video explique que quand tu ecris un prompt de 200 tokens, le modele voit en realite jusqu'a 1 million de tokens (system prompts, fichiers charges, historique, outils). Ton prompt = 0.02% du contexte total. Optimiser ces 200 tokens (D1) c'est bien, mais optimiser les 999,800 autres tokens (D2) c'est la ou se joue la vraie performance.

Toby Litke (CEO Shopify) : "Ca m'a rendu meilleur communicant"

La video cite le CEO de Shopify qui explique que fournir un contexte complet a l'IA l'a force a mieux structurer sa pensee. En preparant des "self-contained problem statements" (enonces de probleme autonomes, sans info externe necessaire), il est devenu meilleur pour communiquer avec ses equipes humaines aussi. La metaphore de la video : c'est comme preparer le bureau de travail avant de commencer - l'agent a tout sous la main.

D3

Intent Engineering

Complete v1

Niveau : Stratégique

Encoder les préférences, priorités et règles de décision pour des actions autonomes alignées.

📌 Implémentation

Fichier unique : .claude/rules/intent.md

  • Taille : 787 tokens
  • Densité : 76% (très efficace)
  • Contenu : Trade-offs, arbres de décision, standards qualité, triggers escalation
  • Auto-chargé : Via .claude/rules/ - effet immédiat

💡 Exemples (issus de la video)

Klarna : 2.3 millions de conversations, la mauvaise metrique

Sans intent engineering : L'agent Klarna a resolu 2.3M de conversations clients en optimisant la vitesse de resolution (fermer le ticket vite). Resultat : crise de confiance, clients insatisfaits, re-embauche d'agents humains.
Avec intent engineering : L'agent aurait du optimiser la satisfaction client, pas la vitesse. L'intent encode : "Satisfaction > Vitesse > Cout". L'agent sait quoi privilegier quand il y a conflit.

Lecon : Le context engineering dit a l'agent quoi savoir. L'intent engineering dit a l'agent quoi vouloir. Sans intent, l'agent optimise pour ce qui est facile a mesurer, pas pour ce qui compte vraiment.

Hierarchie de trade-offs : encoder les priorites

La video explique qu'un agent autonome prend des centaines de micro-decisions. Sans hierarchie explicite, il improvise. Avec une hierarchie comme "Ne pas casser > Fonctionnel > Existant > Simple > Rapide", l'agent sait automatiquement que face a un dilemme, il doit privilegier la stabilite sur la vitesse, l'existant sur le neuf, le fonctionnel sur le parfait. C'est l'equivalent des valeurs d'entreprise, mais pour un agent.

🎯 Hiérarchie de Trade-offs

  1. Ne pas casser > tout le reste
  2. Fonctionnel > parfait
  3. Existant > nouveau
  4. Simple > élégant
  5. Rapide > complet

🌳 Arbres de décision (4)

  • Demander ou Décider : Quand demander confirmation vs agir seul
  • Documenter ou Non : SESSION vs pas de doc
  • Script vs Projet : Où créer un nouveau script
  • FE Comment BV Germany : Créer un commentaire FE ou non

📏 Standards de qualité (3 types)

  • Code : Tests, nommage, structure
  • Docs : Format, placement, headers
  • Scripts : Docstring, chmod +x, error handling

🚨 Triggers d'escalation

  • Incertitude sur impact
  • Décision architecturale majeure
  • Données à supprimer/modifier

🔄 Maintenance

Quand mettre à jour :

  • Nouveau pattern de décision récurrent identifié
  • Une règle ne correspond plus à la réalité
  • Claude demande trop souvent la même chose

D4

Specification Engineering

Complete v1

Niveau : Organisationnel

Écrire des documents complets pour exécution autonome de tâches complexes par agents AI.

💡 Exemple cle (issu de la video)

Personne A vs Personne B : le meme PowerPoint, 10x d'ecart

Personne A (chat + iteration) : Ouvre le chat, demande de faire le PowerPoint, itere sur le resultat, corrige manuellement. 40 minutes. Economie reelle : 50-60%.
Personne B (spec de 11 min) : Ecrit une specification structuree avec les 5 primitives : probleme complet, criteres d'acceptation, contraintes, decomposition, tests. Lance l'agent. 11 minutes. L'agent execute de A a Z sans poser de questions. Productivite : 10x.

Citation video : "The specification done right turns out to be just what clear thinking has always looked like - made explicit because machines don't let us be lazy about it."

📌 Les 5 Primitives

  1. Problem Statement - Auto-cohérent, pas de contexte supplémentaire
  2. Acceptance Criteria - 3 phrases vérifiables
  3. Constraint Architecture - Musts, must-nots, preferences, escalation
  4. Decomposition - Tâches <2h avec I/O clairs
  5. Evaluation Design - Tests de régression avec outputs de référence
